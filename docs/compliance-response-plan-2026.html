Phase 2 · 1~2개월 · 개발 조치
Firestore 복제 범위 최소화 — summary·image 저장 중단

현재 Firestore의 articles 컬렉션에는 summary(기사 요약 텍스트)와 image(이미지 URL)가 복제 저장됩니다. 이 두 필드가 저작물 복제의 핵심 증거가 됩니다.

대응 방향: title + link + publishedAt + categories + publisher만 저장하고, summary와 image는 앱에서 실시간으로 원본 RSS 또는 원본 URL에서 로딩하는 방식으로 전환합니다. 이 구조는:

1Firestore에 저장되는 저작물의 범위가 사실 정보(제목·링크·날짜) 수준으로 축소 → 복제 행위 자체가 대폭 감소
2summary와 image를 원본 서버에서 실시간 로딩 → 원저작권자 서버로 트래픽 유도 → 딥링크 구조와 동일한 법적 효과
3썸네일 이미지 공정이용 판례(대법원 2006) 적용 범위에 부합 — 원본 사이트로의 트래픽 매개체 역할
수정 대상: LifeMash-NewsCollector/news_collector_rss/collector.py
저장 필드에서 제거: summary (2000자 저장 → 삭제), image (이미지 URL → 삭제)
유지: id, publisher, title, link, publishedAt, categories, host, visible, keywords

수정 대상: LifeMash-App ArticleRepositoryImpl / UI
→ 피드 카드에서 summary·image를 원본 RSS/URL에서 클라이언트 로딩으로 전환

Phase 2 · 1개월 · 개발 조치
robots.txt 준수 구현 — 선의 입증 완성

NewsCollector는 이미 ETag·지연·식별 User-Agent를 구현하고 있어 기술적 선의의 기반이 갖춰져 있습니다. 여기에 robots.txt 파싱만 추가하면 "윤리적 크롤링"의 모든 요건을 충족합니다. 나중에 문제가 생겼을 때 "우리는 최선의 기술적 안전장치를 모두 갖췄다"는 방어 논리가 완성됩니다.

# Python 표준 라이브러리로 구현 가능 (외부 의존성 없음)
from urllib.robotparser import RobotFileParser

def is_allowed(feed_url: str, user_agent: str = "LifeMashNewsBot") -> bool:
    rp = RobotFileParser()
    robots_url = f"{urlparse(feed_url).scheme}://{urlparse(feed_url).netloc}/robots.txt"
    rp.set_url(robots_url)
    rp.read()
    return rp.can_fetch(user_agent, feed_url)

robots.txt에서 차단된 피드는 수집에서 제외하고 feeds 컬렉션의 해당 문서에 robotsBlocked: true로 마킹하여 추후 재확인 대상으로 관리합니다.

KPF 빅카인즈 API 전환 — 사업자 등록 시 즉시 실행

사업자 등록이 가능해지는 시점에 즉시 실행할 수 있도록 지금부터 아키텍처를 준비합니다. NewsCollector를 데이터 소스를 추상화한 인터페이스 구조로 리팩터링하여, RSS 수집 로직을 빅카인즈 API 호출로 교체하는 작업이 1~2일 내에 가능하도록 설계합니다.

빅카인즈(BIG KINDS) API: 국내 100여 개 이상 매체의 뉴스 데이터를 합법적으로 분석·활용할 수 있는 NLP 구조화 시스템. 미디어 스타트업 지원사업 선정 시 2년간 무상 제공. 뉴스저작물 저작권·라이선스 문제 완전 해소.
— 2026년 방통위·KPF 미디어 스타트업 지원사업 요강

2지금 할 것: 빅카인즈 API 문서·필드 구조 파악 후 매핑 레이어 설계 문서화

Phase 1 · 즉시 · 개발 조치
최초 실행 동의 화면 구현 — Analytics·Crashlytics opt-in

현재 가장 명백한 위반 상태입니다. Crashlytics가 동의 없이 활성화되어 있습니다. 구현 방향은 단순하게가 최선입니다. 과도한 동의 단계는 사용자 이탈을 유발합니다.

1앱 최초 실행 시 (또는 기존 사용자에게는 다음 업데이트 첫 실행 시) 동의 화면 1장 표시
2동의 내용: "서비스 개선을 위해 앱 사용 통계 및 오류 로그를 수집합니다. [동의] / [동의하지 않음]"
3"동의하지 않음" 선택 시 FirebaseAnalytics.setAnalyticsCollectionEnabled(false) + FirebaseCrashlytics.setCrashlyticsCollectionEnabled(false) 호출 후 앱 진행
4동의 여부를 DataStore에 저장, 앱 재실행 시 중복 표시 방지
5설정 화면에 "개인정보 수집 설정" 토글 추가 — 사후 변경 가능하도록
// SDK 초기화를 동의 이전으로 앞당기지 말 것
// Application.onCreate() 에서 동의 상태 확인 후 조건부 초기화
if (isAnalyticsConsented) {
    FirebaseAnalytics.getInstance(this).setAnalyticsCollectionEnabled(true)
    FirebaseCrashlytics.getInstance().setCrashlyticsCollectionEnabled(true)
}

읽기 이력 자동 파기 — 90일 보유 후 삭제

Room DB의 reading_history 테이블은 현재 무기한 저장됩니다. 90일 보유 정책을 수립하고 자동 파기 로직을 구현합니다.

// WorkManager 주기 작업으로 90일 초과 레코드 삭제
@Query("DELETE FROM reading_history WHERE readAt < :cutoffTime")
suspend fun deleteOlderThan(cutoffTime: Long)

처리방침의 보유기간 항목도 "읽음 기록: 90일 후 자동 파기"로 업데이트가 필요합니다.

사업자 등록 전까지의 포지셔닝: "수익 없는 개인 비상업적 프로젝트"를 이용약관에 명문화하고 유지합니다. 이 포지셔닝은 저작권·부정경쟁방지법 제재의 핵심 요건인 "영리 목적"을 충족하지 않으므로, 법적 공세의 1차 표적에서 벗어납니다. 단, 이것은 면죄부가 아닌 리스크 경감 수단임을 인지해야 합니다.

가장 중요한 단일 조치: Firestore에서 summary·image 복제를 중단하고 클라이언트가 원본 소스에서 직접 로딩하는 구조로 전환하는 것입니다. 이 하나의 변경이 뉴스 저작물 복제의 핵심 증거를 제거하고, 동시에 원저작권자 서버로 트래픽을 유도하는 구조를 만들어냅니다.
